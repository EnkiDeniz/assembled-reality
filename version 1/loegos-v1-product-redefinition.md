# Lœgos v1 Product Redefinition

## Purpose

This document resets the product definition for Lœgos v1.

The purpose of the system has not changed.

What has changed is the product form.

Lœgos is no longer best described as:

- a source-to-seed workspace
- a visible AI document tool
- a language workbench alone

It is better described as:

`a visible assembly system that helps a human turn an aim into reality by capturing sources, structuring them into Lœgos, choosing bounded moves, testing them in the world, and preserving the receipts of what actually held`

This document is the clean-slate definition for that product.

## What stays constant

The invariant purpose is:

- help a human move from direction toward reality
- prevent fake convergence
- make structure legible
- preserve the difference between witness, interpretation, and proof
- let the human learn the language rather than be hidden behind it

The user should be able to see how an idea was assembled from:

- a seed in the mind
- to words
- to sentences
- to documents
- to structured language
- to actions
- to receipts
- to reality

## What changed

The system has shifted away from being primarily a shell made of screens and phases.

It now looks more clearly like a product with four deep layers:

1. `Capture`
- anything relevant to the aim can enter the box

2. `Assemble`
- sources are normalized, broken into meaningful units, compared, and converted into Lœgos

3. `Act`
- one bounded move is chosen and tested

4. `Realize`
- receipts accumulate and the once-abstract aim becomes materially real

The old shell over-centered views and UI surfaces.

The new product must center:

- assembly
- traceability
- alignment
- bounded movement
- returned proof

## The product in one sentence

Lœgos v1 is a system for making assembly visible, teachable, and reality-bearing.

## The backbone

Three layers define the product and should remain distinct:

### Product

`state -> move -> test -> receipt -> update -> seal / flag / stop -> return`

This is the outer loop the human is actually trying to complete.

### Language

`declare -> capture -> observe -> weld -> move -> test -> receipt -> seal`

This is the assembly grammar that turns witness into operable structure.

### Agents

`rate -> tighten weakest axis -> rerate -> seal / flag / stop`

This is the governed loop for Seven and any future bounded intelligence inside the system.

These three layers support each other, but they should not collapse into one another.

## The key difference from the chat-first sibling product

The sibling product helps the user think through chat.

That can be useful, but it hides too much:

- the language being spoken
- how source material was interpreted
- how interpretation became structure
- how structure became action
- how action became proof

Lœgos v1 exists to do the opposite.

It should let the user:

- drop in source material quickly
- hear it
- inspect it
- see how it was converted into Lœgos
- learn the language over time
- see which parts actually advanced the aim
- trace how a real-world result was assembled step by step

The promise is not merely:

`the AI helped me`

The promise is:

`I can see how this became true`

## Product thesis

Words are like legos.

Each document contains blocks of meaning.

Some blocks matter.
Some repeat.
Some ring a bell.
Some support the aim.
Some distract.
Some remain noise until later.

The job of Lœgos is to help the human:

- capture these blocks
- hear them
- mark the important ones
- turn them into explicit structure
- decide what to do next
- test the next move
- keep the proof trail

That is how an imagined farmhouse in upstate New York can become a real farmhouse:

- image
- note
- plan
- budget
- research
- land search
- home design
- permit steps
- financing actions
- receipts
- reality

Lœgos should make that assembly trace visible.

## Core product objects

### 1. Aim

The declared direction.

This is not a vague aspiration field.

It is the thing the box is trying to advance.

### 2. Box

The container for one assembly space around an aim.

A box holds:

- the aim
- all witness material
- the active Lœgos structure
- moves and tests
- receipts
- current state

### 3. Witness

Any source object brought into the box and normalized into a readable markdown-backed form.

Witness may begin as:

- markdown
- pasted text
- word document
- pdf
- image
- audio
- link

Witness is the source-facing truth object.

### 4. Block

A meaningful unit inside a witness or Lœgos document.

Blocks are the first manipulable units of assembly.

They may later be re-typed, grouped, staged, compared, or advanced.

### 5. Lœgos language

The explicit structured form derived from witness and then actively edited.

This is where the system becomes legible as language rather than raw content.

### 6. Move

One bounded next action intended to shrink the gap between aim and reality.

### 7. Test

A small reality-contact step that can confirm, weaken, or refute the move.

### 8. Receipt

The returned proof object that records what happened and whether the state changed.

### 9. State

The current alignment condition of the box.

For v1, this is best understood as a live relation between:

- `△ Aim`
- `□ Reality`
- `○ Story`

## The core loop

The top-level loop of the product is:

1. declare the aim
2. capture witness material
3. listen, read, and mark what matters
4. convert selected structure into Lœgos
5. compare witness and Lœgos
6. choose one bounded move
7. test it in reality
8. record the receipt
9. update the state
10. seal, flag, or stop honestly

This is the primary product loop.

The workbench, compare, Operate, and receipts all support this loop.

## Navigation language

The design language already points in the right direction.

The top-level navigation should be simple, calm, and language-native.

For v1, the primary verbs should be:

### `△ AIM — declare`

Direction, declaration, next move.

This is where the box states what it is trying to become true.

### `□ REALITY — capture`

Capture, listen, inspect what is here.

This is where witness material enters and where the user spends most of their mobile time.

### `œ WELD — rewrite`

Stage, rewrite, operate the object.

This is where witness becomes Lœgos, where structure is compared, shaped, and prepared for action.

### `⟁ SEAL — review`

Review, seal, share the proof.

This is where the returned evidence and receipts are judged, not merely displayed.

These are not just labels.

They are the stable product grammar.

## Platform split

The product should behave differently on mobile and desktop because the dominant jobs are different.

### Mobile is primarily for capture and listening

On mobile, the main user behaviors are likely to be:

- add a source quickly
- read it
- listen to it
- follow along while listening
- select a block or section
- say: this matters, this rings, this should advance
- lightly annotate or mark importance
- resume later

Mobile should optimize for:

- speed
- calm reading
- one-hand listening flow
- easy block marking
- no confusion about object state

Mobile should not be the place for most detailed structural work.

### Desktop is primarily for assembly and judgment

On desktop, the main user behaviors are:

- inspect witness deeply
- convert witness into Lœgos
- compare witness and active structure
- edit the Lœgos form
- inspect the game state
- run Operate
- judge override
- draft and review receipts

Desktop is where the deeper assembly work happens.

## Mobile-first critical loop

The most important mobile loop for v1 is:

1. open the box
2. add a source fast
3. normalize it into witness
4. listen to it in full or in sections
5. have the text follow the audio
6. mark one or more blocks as important to the aim
7. save that marked state back into the box

This loop must be excellent.

If this loop is weak, the product loses one of its strongest real-world advantages.

## Desktop-first critical loop

The most important desktop loop for v1 is:

1. open the box
2. inspect the current witness material
3. compile witness into Lœgos
4. compare witness and Lœgos side by side
5. understand what changed and why
6. edit the Lœgos structure
7. choose one next move
8. test and log the receipt
9. update the state honestly

## Witness logic

Witness is the immutable source truth inside compare.

Outside compare, witness can still be edited at notes-app level if the user is cleaning, correcting, or extending the source object.

But witness and Lœgos are not the same thing.

The rule is:

- witness edits do not silently mutate the active Lœgos object
- witness edits require explicit compile or recompile before the language changes

This keeps the assembly trace honest.

## Lœgos logic

Lœgos is the active structured working object.

It is where witness becomes legible as explicit semantic structure.

The user must be able to:

- inspect the type of a block
- understand why it was shaped that way
- rewrite it
- stage it
- compare it to witness
- see whether it supports the aim
- see whether it is weak, repetitive, unsupported, stale, or contradicted

The goal is not only that the AI can speak Lœgos.

The goal is that the human can learn to read it and eventually use it with growing fluency.

## Listening logic

Listening is not a side feature.

Listening is one of the primary sensing mechanisms of the system.

The v1 listening rules are:

- any witness object should be playable
- playback should follow the text visibly
- playback should be resumable
- the user should be able to move by section or block
- the user should be able to mark a block while listening
- the marked block should be attachable to the aim as something to advance

This is especially important on mobile.

## Marking and advancement logic

The user needs a simple action that means:

`this block matters to the aim`

This is not yet the same as seal, proof, or final structure.

It is a first advancement signal.

For v1, marked witness blocks should be able to become:

- candidates for compile
- candidates for staging
- candidates for next move discussion
- candidates for future receipt relevance

This creates a bridge between capture and assembly without demanding full workbench behavior on mobile.

## Compare logic

Compare is not optional polish.

It is the place where the user learns:

- what the witness said
- what the Lœgos form says
- why the transformation happened
- what became more operable

The compare rule is:

- witness remains visible
- Lœgos remains editable as the active structure
- the user can tell what changed and why

Without this, the conversion layer becomes arbitrary and unteachable.

## Operate logic

Operate should remain downstream of the active Lœgos object, not a direct witness action.

Operate exists to inspect the current structured state, not to replace the state.

Its outputs should inform:

- diagnostics
- gap awareness
- next move quality
- trust
- contradiction
- seal readiness

Operate is not generic chat.

It is a bounded read of the current active structure.

## Seven logic

Seven should not be a floating assistant.

Seven should be the governed operator inside the alignment game.

Its role in v1 is:

- help rate the state
- identify the gap
- recommend one bounded move
- help phrase one test
- help interpret the returned receipt
- recommend `seal`, `flag`, or `stop`

Seven should not be allowed to:

- refine indefinitely
- pretend coherence equals convergence
- grant seal unilaterally
- erase the distinction between source, language, and proof

## Receipt logic

Receipts are not export artifacts.

They are the runtime proof objects of the system.

Each meaningful cycle should be able to preserve:

- the state before
- the move
- the test
- the returned signal
- the state after
- whether the result was:
  - sealed
  - flagged
  - stopped

Receipts are how assembly becomes visible over time.

## Feature list for v1

### Must have

- authentication and box persistence
- fast source intake
- normalization into markdown-backed witness objects
- calm witness reading
- notes-level witness editing
- strong listening on witness objects
- text-follow during playback
- block or section marking while listening
- explicit compile from witness to Lœgos
- meaningful compare
- editable Lœgos structure
- governed Seven behavior
- one-move / one-test logic
- receipt creation and review
- honest seal / flag / stop states
- clear return to box state

### Should have

- image import that becomes usable witness
- lightweight annotations on witness blocks
- stale detection after witness changes
- explicit recompile
- line or block-level rationale for Lœgos conversion
- visibility into repeated or resonant blocks
- clear mobile resume state

### Not the center of v1

- broad dashboarding
- too many modes
- endless tabs
- generic AI conversation
- feature-heavy but conceptually muddy panels

## How things relate to each other

The relationship logic for v1 is:

- an `Aim` gives a box its direction
- a `Box` contains all assembly objects for that direction
- `Witness` objects are captured reality-facing material
- witness is broken into `Blocks`
- selected or marked blocks can be advanced
- `Lœgos` is the structured operational form built from witness
- `Operate` reads the active Lœgos state
- `Seven` helps govern the bounded convergence loop
- a `Move` and `Test` touch reality
- a `Receipt` records what returned
- `Seal` only happens when the gap is honestly closed

## The farmhouse truth test

The product should eventually let a user trace a real outcome like a farmhouse in upstate New York from its earliest seed onward.

That means the system should make visible:

- the first faint declaration
- the images and notes that entered the box
- the research and witness material
- the blocks that kept ringing
- the Lœgos structures that clarified the situation
- the concrete moves that were taken
- the tests and returned proof
- the receipts that accumulated
- the point at which the idea became reality

If the system cannot make that trace legible, then it is not yet fulfilling its deeper purpose.

## Final definition

Lœgos v1 is a calm assembly system for turning aims into reality through visible source capture, listening, structured language, bounded moves, governed AI, and receipts that preserve how truth was actually assembled.
