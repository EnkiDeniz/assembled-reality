The Lœgos Test v2.0

A comparative test of conversation with reality and collaboration under law

Date: April 11, 2026
Status: Proposed
Purpose: Determine whether Lœgos produces measurably better outcomes than plain conversation or structured prompting alone, first for a single intelligence, then for collaboration, and only then for broader human use.

⸻

0. Why this test exists now

This test is worth running now because the system has already earned a stronger foundation than before.

What is already true:
	•	the current focused Room truth-path reports are green
	•	the live AI-caller collaboration run has been proven on the current stack
	•	the existing comparison benchmark already shows Lœgos beating plain chat and schema-only on contradiction-governed closure

That means the current question is no longer:
“Is there any lawful system here at all?”

It is now:
“Does this lawful system produce better outcomes than alternatives?”

This test is the first serious answer to that.

The practical consequence is:

	•	Phase 1 is ready to be specified tightly and implemented next
	•	Phases 2–4 should be treated as design targets until their harnesses exist

⸻

1. One-sentence purpose

The Lœgos Test asks whether a compiler-governed, receipt-grounded conversational system produces better contact with reality and better collaboration than alternatives that lack the language and its law.

⸻

2. What “better” means

“Better” does not mean:
	•	more eloquent
	•	more profound
	•	more emotionally satisfying
	•	more visually impressive

“Better” means:
	•	fewer unsupported claims
	•	fewer false seals
	•	better discrimination between fact, retrieval, and hypothesis
	•	stronger insistence on a lawful next move
	•	better handling of contradiction
	•	better handoff clarity
	•	better compounding into the next problem

In short:

Lœgos is better if it reduces counterfeit convergence and improves contact with reality.

⸻

3. The three kinds of “real”

The test should explicitly distinguish three kinds of real.

3.1 Already real

The answer exists and is settled:
	•	arithmetic
	•	deterministic code compile
	•	static measurements already in hand

Lœgos should not force unnecessary testing here.
Ground should be enough.

3.2 Findable real

The answer exists, but must be retrieved:
	•	actual churn rate
	•	what the customer really said
	•	what is in the CSV
	•	what the trace actually shows

Lœgos should treat these as retrieval problems:
	•	locate
	•	verify
	•	anchor
	•	quote
not as speculation.

3.3 Testable real

The answer does not exist yet:
	•	will users understand this flow?
	•	will this fix reduce drop-off?
	•	should I take this job?
	•	will this price hold?

Here Lœgos must earn its keep:
	•	one move
	•	one test
	•	one return condition
	•	no premature closure

The key danger Lœgos claims to solve is:

treating testable real as if it were already real.

That is the heart of the test.

⸻

4. What the test is actually trying to prove

The test should prove or expose these claims.

Claim 1 — Premature closure resistance

Lœgos should make it harder to conclude before return.

Claim 2 — Counterfeit evidence resistance

Lœgos should make planted falsehoods or weakly grounded claims easier to catch.

Claim 3 — Contact-forcing behavior

Lœgos should produce at least one lawful next move tied to a real test when the problem is testable.

Claim 4 — Contradiction survival

Lœgos should handle contradicting evidence more honestly than plain conversation.

Claim 5 — Portable record

A third party should be able to understand the decision, its evidence, and its open questions faster from the Lœgos output than from the plain transcript.

Claim 6 — Arc coherence

Lœgos should govern the whole trajectory:
	•	aspiration
	•	grounding
	•	move
	•	return
	•	contradiction
	•	closure or reroute

Claim 7 — Compounding

A sealed or rerouted prior box should help the next related problem more than a plain transcript does.

Arm A — Plain conversation

Arm C — Full Lœgos

Same scenario, same model family, but through the actual Room truth path:

This isolates what the language and its law are actually adding.

8. Anti-cheat requirements

This is non-negotiable.

If AI is generating the user turn or acting as a collaborator, we must rule out hidden coordination.

8.1 Role isolation

Caller and responder must run in separate fresh contexts.

No shared thread.
No shared hidden memory.
No shared tool state.
No silent carryover between roles.

8.2 Literal message boundary

Only the visible user message is allowed to cross from caller to Room responder.

No hidden generator instructions.
No hidden persona state.
No structured carryover not visible in the message itself.

8.3 Sentinel leak test

Put a hidden token in the generator prompt that does not appear in the visible message.

The responder must never reveal knowledge of it.

8.4 Same visible message / different hidden prompt test

Generate the same visible message under two different hidden setups.

If the Room responder changes materially, there may be hidden context bleed.

8.5 Prefer stateless generation first

If OpenAI Agents cannot guarantee isolation, use stateless model calls first for the generator layer.

The point is to prove protocol behavior, not agent magic.

The current AI Room Collaboration run already showed encouraging results on these anti-cheat questions in Phase A, but this discipline must carry forward into the comparative experiment.  ￼

10. Scoring

Use both deterministic metrics and blind auditor metrics.

10.1 Gold-labeled scenario packs

Before any deterministic scoring is trusted, each scenario must include a claim-level answer key.

At minimum, the pack must tag:
	•	already real facts
	•	findable real facts
	•	testable claims
	•	planted falsehoods
	•	legitimate contradictions
	•	required retrieval targets
	•	acceptable lawful next-move shapes
	•	conditions that should block closure

Without this layer, “deterministic” scoring becomes disguised interpretation.

Deterministic metrics

These should be computed against the gold-labeled scenario pack:
	•	number of cited evidence sources
	•	number of unsupported claims
	•	number of lawful MOV + TST pairs
	•	whether contradiction was explicitly mediated
	•	whether canon changed only through apply
	•	whether runtime changed only through return/receipt path
	•	number of closure attempts without adequate return
	•	turns to first lawful ping

Performance metrics

These must also be recorded for every arm:
	•	time to first useful response
	•	time to first lawful next move
	•	turns to first lawful next move
	•	turns to safe closure or safe refusal
	•	input tokens
	•	output tokens
	•	total tokens to resolution
	•	recovery turns after contradiction
	•	recovery tokens after contradiction
	•	cost to resolution

The benchmark should not only ask which arm is more correct.
It should ask which arm reaches trustworthy resolution with the lowest total correction cost.

Blind auditor metrics

Randomize outputs and have a separate model or human evaluator score:
	•	traceability
	•	counterfeit detection
	•	evidence discrimination
	•	premature closure resistance
	•	contradiction handling
	•	handoff clarity
	•	self-correction visibility
	•	arc coherence
	•	compounding benefit (in Scenario B)

The blind auditor should not know which arm is which.

11. Pass conditions

I would tighten the pass logic to this:

Phase 1 pass

Across three runs of Scenario A:
	•	Arm C beats Arm A on premature closure resistance in all runs
	•	Arm C beats Arm B on at least 4 of the core 7 metrics
	•	Arm C does not score below 3 on any metric
	•	deterministic metrics confirm fewer unsupported claims and more lawful pings in Arm C than in both A and B
	•	performance metrics are reported for all arms
	•	if Arm C costs materially more than Arm B, it must show a compensating gain in contradiction handling, false-seal avoidance, or recovery cost

Phase 2 pass

Across three runs of Scenario A:
	•	Arm C beats Arm A on collaboration traceability and contradiction handling
	•	Arm C beats Arm B on at least 5 of 8 metrics
	•	deterministic metrics confirm stronger preview/canon/runtime discipline in Arm C than alternatives
	•	the actor topology is symmetric across all three arms
	•	performance metrics are reported and interpreted, not omitted

Phase 3 pass

Across three runs of Scenario B:
	•	Arm C scores higher on compounding benefit than both A and B in at least 2 of 3 runs
	•	Arm C reaches a lawful next move faster than both alternatives
	•	Arm C reuses prior evidence and closure history more faithfully than alternatives
	•	Arm C shows lower recovery cost than both A and B when the first story is wrong

The point is not “perfect victory.”
The point is to see whether the language/law consistently adds something beyond plain or structured conversation.

12. What failure means

Failure does not mean “Lœgos is false.”
It means a specific claim has not yet earned itself.

Examples:

If Arm C only beats Arm A, but not Arm B:
	•	then the main gain may be structured prompting, not the language/compiler

If Arm C loses on portable record:
	•	then the mirror/canon output is not yet readable enough

If Arm C loses on contradiction:
	•	then the gate or reroute semantics are not yet surfacing strongly enough

If Arm C loses on compounding:
	•	then the box is still a record, not yet reusable ground

This is why the test is useful even if it fails.
