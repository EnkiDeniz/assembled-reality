# Final Next-Phase Proposal

Date: April 11, 2026  
Status: Proposed phase plan  
Author: Engineer  
Purpose: Define the final detailed next-step sequence from the current built product to the next trustworthy human-drive phase without overengineering.

---

## 1. Executive Summary

We should treat the next phase as a disciplined convergence phase, not an invention phase.

We already have:

- a real law kernel
- a real visible working echo
- a real short-loop benchmark
- a real Drive Tape replay instrument with visible source classification
- a real signal that sighted Lœgos now beats blindfolded

We do **not** yet have:

- a fully language-faithful working echo
- a human-validated steering loop
- a reverse-pass legibility instrument in product form
- proof travel integrated into the surfaced read
- visible stabilization as a first-class measured relation

So the next phase should do this in order:

1. prove reverse-pass legibility cheaply
2. strengthen the current short loop where it is still weak
3. add explicit language-fidelity testing
4. run a real human drive
5. compare clearer rendering against bounded advisory
6. add proof travel only if the short loop and human drive justify it
7. only then stretch into longer-loop testing

The meta-rule for the whole phase is:

**surface what the language already knows before adding more machinery.**

And every major next-step review in this phase should run a double check:

1. does this fit Braided Emergence?
2. what is the Elden Ring equivalent?

Braided Emergence checks whether the structure is lawful.
The Elden Ring check asks whether the thing is actually playable:

- what is the boss?
- what counts as death / failed run?
- what is the bonfire?
- what is the lore on the wall?
- what gets sharper after return?

## 3. The Main Goal Of The Next Phase

By the end of this phase, we should be able to say:

1. the visible working echo is stronger, more artifact-faithful, and more useful in fog
2. a human can steer better with it than with chat alone
3. we can see the survival structure of a real interaction through replay, not just through prose review
4. we can inspect where coherence and convergence are or are not being held together
5. we still have not added any unnecessary second meaning engine

That is the success condition.

## 4. The Main Risks To Avoid

### 4.1 Overengineering risk

We add:

- advisory reasoning
- proof machinery
- richer trace UI

before the current loop is clearly surfaced and human-drivable.

### 4.2 Parallel meaning risk

The machine starts inventing:

- aim
- deciding split
- closure
- truth status

instead of surfacing what Lœgos, return, and receipts already know.

### 4.3 Pretty-but-useless risk

We build:

- nicer replay visuals
- more beautiful prose
- more interface layers

without improving steering, truth legibility, or proof travel.

### 4.4 Premature long-loop risk

We move to Test Drive III before:

- short-loop behavior is stronger
- human drive says the hinge is real
- the language-fidelity lane is mature enough to trust

## 5. The Phase Sequence

## Stage 0: Reverse-Pass Legibility Diagnosis

### Goal

Prove that replaying the loop helps us see the system more truthfully than prose review alone.

More specifically:

- make the current braid visible enough to inspect
- use replay to expose the early stabilizer function before we try to formalize it as a runtime object

### Use

Use the current `Drive Tape v0` implementation and visible source classification on the latest Test Drive II corpus:

- no live mode
- no new authority path
- no editing
- no advisory overlays

### Test

Run the tape against:

1. `contradictory_return_journey`
2. `working_echo_correction`
3. `no_move_yet`

And answer:

- what was only spoken?
- what became visible structure?
- what bent after return?
- what weakened?
- what survived?
- where did re-aim happen?
- where does the tape show coherence and convergence being held together, and where do they drift apart?
- which fields are return-authored versus assistant-authored?
- did correction alter the next proposal, or only the narrative around it?

### Exit criteria

Stage 0 is successful if:

- the tape makes the deciding split easier to see than prose review alone
- the tape makes the `contradictory_return_journey` authorship gap obvious
- the tape reveals at least one meaningful signal-survival pattern we would have missed in chat review
- the tape gives us a clearer read on stabilization than the current score table alone

### Stop condition

If the tape is merely interesting-looking and does not sharpen diagnosis, stop expanding it.

## Stage 1: Strengthen The Short Loop

### Goal

Make the current working echo more truthful and more useful before adding new central machinery.

### Build priorities

1. strengthen `aim`
   - make it first-class on the surface
   - make it clearly revisable
   - source it from lawful structure where possible

2. strengthen `whatWouldDecideIt`
   - keep it sharp, discriminating, and non-generic
   - prefer specific missing witness or divergence questions over vague “more info”
   - in return-heavy cases, let the split be rewritten by return rather than merely paraphrased from the assistant answer

3. harden `supports / weakens / missing`
   - better evidence handles
   - less gist
   - clearer provenance

4. preserve the `no_move_yet` gain
   - keep “why not move yet” explicit
   - keep the one next witness or split visible
   - do not let later changes collapse it back into generic fog

5. reduce heuristic bridge authorship
   - audit where `workingEcho` still depends on prose-derived inference
   - shift fields toward lawful artifact or runtime sourcing where possible
   - prioritize `aim` and `whatWouldDecideIt` in return-heavy cases

6. strengthen visible stabilization cues without turning `stabilizer` into a heavy new UI noun
   - make return bend unmistakable
   - make weak vs strong evidence carry-forward clearer
   - make local closure vs reroute easier to read

### Tests

- keep law tests green
- keep Test Drive II green
- add explicit artifact-fidelity checks on `workingEcho`

### Exit criteria

Stage 1 is successful if:

- `contradictory_return_journey` materially improves
- `no_move_yet` stays strong
- `aim` is visible enough to steer with
- major `workingEcho` fields have clearer source identity
- Test Drive II remains valid and competitive

## Stage 2: Add The Language-Fidelity Lane

### Goal

Measure whether the machine is surfacing what Lœgos already knows or inventing parallel meaning.

### Build

For each important surfaced field, classify its source:

- lawful artifact
- bounded provisional state
- runtime return
- bounded advisory
- heuristic bridge logic

At minimum, cover:

- `aim`
- `whatWouldDecideIt`
- `candidateMove`
- `supports / weakens / missing`
- `returnDelta`

### Test questions

- does the lawful artifact already know this?
- if yes, are we surfacing that directly?
- if not, is the bridge logic clearly marked provisional?
- is the machine adding new meaning that the thread never really earned?

### Exit criteria

Stage 2 is successful if:

- we can explain the source of every major surfaced field
- artifact-backed fields are preferred over heuristics
- the heuristic layer is clearly bounded rather than quietly authoritative
- we can explain what is doing stabilizer work today:
  - law
  - return
  - benchmark / measurement
  - human interpretation
  - surfaced trace

---

## Stage 3: Human Drive

### Goal

Run a real human steering check before adding more machinery.

### What to compare

1. chat only
2. current working echo
3. optionally a simple board control

### Questions

- can the human identify the current aim?
- can they see what supports the read?
- can they see what weakens the easy story?
- can they identify the next witness?
- can they tell when not to move yet?
- can they see return re-aim the work?
- can they feel when the system is holding versus merely sounding coherent?
- can they tell what the boss is?
- can they tell where the bonfire is?
- can they tell what gets sharper after a failed pass?

### Success criteria

Stage 3 is successful if humans materially:

- steer better than from chat alone
- choose sharper missing witnesses
- understand return bend and re-aim more clearly

### Failure interpretation

If humans do **not** steer better, the missing value is still rendering, not advisory intelligence.

---

## Stage 4: Rendering vs Advisory

### Goal

Answer whether the next gains come from better surfacing or from bounded advisory.

### Build

Run a bounded comparison:

1. clearer rendering of existing Lœgos structure
2. the same surface plus bounded Shape Library advisory

The stabilizer rule here is important:

- do not treat `stabilizer` as a new oracle
- first ask whether clearer rendering already makes the stabilizing relation visible enough

### Shape Library burden of proof

Shape Library earns a larger role only if it improves one or more of:

- deciding split
- receipt condition
- disconfirmation line

And it must beat rendering alone.

### Exit criteria

Stage 4 is successful if we can say clearly:

- rendering alone recovers most of the value

or:

- bounded advisory adds value beyond rendering alone

Without that answer, Shape Library should remain secondary.

---

## Stage 5: Proof Travel

### Goal

Add proof status to the surfaced loop without turning it into another chatter layer.

### Build

Add bounded receipt-backed provenance to the working echo:

- draft proof
- sealed proof
- verified proof
- latest receipted return

Do **not** add raw GetReceipts objects or another thought layer.

### Test

Measure whether proof status improves:

- trust
- handoff
- operator confidence in the read
- ability to reconstruct why the state should be believed

### Exit criteria

Stage 5 is successful if proof status makes the read more portable and trustworthy without making the surface noisier or more bureaucratic.

---

## Stage 6: Longer Loop Only After The Above

### Goal

Only then expand into Test Drive III and broader longer-loop questions.

### Preconditions

Do not start this stage until:

- short loop is stronger
- language-fidelity lane is in place
- reverse-pass replay proved useful
- at least one human drive succeeded
- proof status is bounded and readable enough to matter

### Then test

- move validity across later state
- return honesty
- re-aim honesty
- handoff integrity
- proof travel

## 8. Decision Gates

At the end of each stage, ask:

### Gate A

Did this improve steering feel?

### Gate B

Did this improve truth legibility?

### Gate C

Did this reduce or increase parallel machine-authored meaning?

### Gate D

Did this earn its weight?

### Gate E

Does it pass both:

- the Braided Emergence law check
- the Elden Ring playability check

If a change fails those gates, it should not become central.
