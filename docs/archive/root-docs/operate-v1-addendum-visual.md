# Operate V1 Addendum: Visual Source Handling

## Problem

The current spec treats image upload as OCR — extract text, produce .md, move on. That works for a photo of a whiteboard or a scanned document. It does not work for most of what users will actually drop into a box.

When someone uploads a photo of an upstate New York ranch, a robot, a Star Wars still, and a satellite view of available land, they are not providing text. They are providing signal. Each image carries intent, reference, evidence, or aesthetic — and the system needs to read that signal the same way it reads text sources.

Images are first-class sources, not OCR targets.

---

## How It Works

### 1. Image enters the box

When a user uploads an image, the system does not default to OCR. It runs a visual description step first.

The description step produces a structured .md source with three fields:

```markdown
# Visual Source: [filename or user label]

## Description
[What the image contains. Plain language. 2–5 sentences.]

## Signal
[What this image likely means in the context of this box. Why did the user add it? 1–2 sentences.]

## Shape
[△ or □ or ○] — [short rationale]
```

The description is generated by the model. The signal and shape are inferred from the image content combined with whatever else is already in the box. Context matters — the same image of a sheep farm means something different in a box about agriculture versus a box about photography.

### 2. Shape classification for images

Images tend to cluster differently than text:

| Image type | Typical shape | Why |
|---|---|---|
| Aspirational / mood / vision | △ | The user is collecting what they want to build toward. Not evidence — aim. |
| Reference / aesthetic / style | ○ | Story, vibe, world-building. "This is the feeling." |
| Photo of a real place, object, or document | □ | Evidence. This exists. This is real. |
| Screenshot of data, a dashboard, a receipt | □ | Measurement. Ground. |
| Sketch, diagram, wireframe | △ or ○ | Depends on whether it's proposing (△) or connecting (○). |
| Photo of a person, team, or event | ○ or □ | Story (context) or evidence (this happened). |

These are defaults. The system can infer more precisely using the box context. A photo of land in a box called "Farm Project" is □ — real place, real constraint. The same photo in a box called "Dream Board" is △ — aspiration.

### 3. How images feed into Operate

Once described and classified, the image source is a .md file like any other. Operate reads it in Step 2 (shape-read) alongside text sources. The image's description and shape classification participate in:

- The box shape distribution (images shift △□○ balance)
- The gradient assessment (a box full of aspirational images is gradient 1–2, not 5–6)
- The compression step (image descriptions are compressed into the three operator sentences alongside text)
- The level-read (a photo with GPS metadata and timestamp can be L2; an aspirational image with no verifiable content is L1)

### 4. What Operate does with a mixed box

Example box: "Tech Farm — Upstate NY"

Sources:
- 3 photos of upstate ranch properties (real estate listings)
- 1 photo of a robotic milking system
- 1 Star Wars concept art image (moisture farm aesthetic)
- 1 text note: "I want to build a technology-integrated farm within 2 hours of NYC"
- 1 link: article about agri-tech startups in the Hudson Valley

After visual description and shape-read:

```
Sources: 7 (5 image, 1 text, 1 link)

Shape distribution:
  △ 52%  (ranch photos as aspiration, robot as vision, note as aim)
  □ 18%  (article as evidence, ranch listing prices as ground)
  ○ 30%  (Star Wars reference as aesthetic/story)

Gradient: 2 (warming — material gathering, no commitment yet)

Trust profile:
  L1: 70%  (most images are aspirational, note is self-declared)
  L2: 30%  (article exists, listing photos have source URLs)
```

Operate compresses to:

> **△** You aim to build a technology-integrated farm property within two hours of NYC, referencing automated systems and a rugged, functional aesthetic.
> **□** Three real estate listings exist in the reference set. One industry article covers the Hudson Valley agri-tech corridor. No budget, no zoning research, no site visits.
> **○** The box has strong visual direction but no ground contact — the gap is between the vision (clear) and the first concrete step (absent).

> **Gradient:** 2
> **Floor:** L1 — mostly aspirational, self-reported
> **Ceiling:** L2 — some sources have verifiable URLs
> **Next move:** Pick one property listing. Research its zoning status. That's the first □ source that moves this box from dream to directed.

That's the output. The user dropped in five images and two text sources. The system read all of them, classified them, and produced a diagnosis that's honest about where the box is and what the next move looks like.

## Implementation Notes

### Visual description model

- Use the existing model's vision capability for description generation.
- The description prompt should be grounded: describe what you see, not what you think the user wants to hear. Honest description, not enthusiastic interpretation.
- Include observable details: setting, objects, text visible in the image, apparent condition, scale cues.
- Keep descriptions to 2–5 sentences. This is a source, not an essay.

### Shape inference

- Shape classification uses both the image description and the existing box context.
- If the box is empty (first source), default to △ for aspirational images, □ for documentary images.
- If the box has existing sources, classify relative to the existing shape distribution and apparent box purpose.
- Shape classification is a soft label, not a hard gate. It informs Operate's compression but doesn't lock the image into a category.

### Level assignment for images

| Image characteristic | Level |
|---|---|
| No metadata, no verifiable source, no context | L1 |
| Has EXIF data, GPS coordinates, timestamp, or a verifiable source URL | L2 |
| Seven has audited: metadata is consistent, location is plausible, timeline checks out | L3 |

Images cannot reach L4+ on their own in V1. Pattern and network verification require human confirmation across time.

### What not to do

- Do not OCR every image by default. OCR is a fallback for images that are clearly text-heavy (documents, whiteboards, screenshots of text). The system should detect whether an image is text-primary or visual-primary and route accordingly.
- Do not discard images that have "no useful information." An image of a Star Wars moisture farm in a box about building a tech farm is a high-signal △○ source. It tells you what the user wants to feel like. That's not noise — that's aesthetic aim.
- Do not over-describe. A photo of rolling green hills doesn't need five sentences about color temperature and atmospheric perspective. "Rolling hills, open pasture, upstate New York light, barn visible in middle distance" is enough. The compression step handles the rest.
- Do not hallucinate details not visible in the image. If you can't tell the location, say "location unidentified." The system's credibility depends on honest description. An L1 image that says "location unknown" is better than an L2 image that fabricates a GPS coordinate.

## Changes to Operate V1 Spec

### Step 1 — Standardize

Replace the current image row:

| Source Type | Conversion | Result |
|---|---|---|
| Image (text-primary: document, whiteboard, screenshot of text) | OCR + structure | .md |
| Image (visual-primary: photo, reference, sketch, mood) | Visual description + shape classification + level assignment | .md |

The system auto-detects which path to use. Text-primary images go through OCR. Visual-primary images go through description. Both produce .md sources that enter the same pipeline.

### Step 2 — Shape-Read

No change needed. Image-derived .md sources participate in the shape-read like any other source. Their pre-assigned shape classification seeds the read but can be adjusted by Operate if box context suggests a different classification.

### Step 3 — Compress

No change needed. Image descriptions are part of the corpus that gets compressed into the three operator sentences. The model should draw on visual sources the same way it draws on text sources.

### Result Surface

Add to the included material summary:
- Source count should distinguish text and image sources: "3 text sources, 5 image sources"
- If all sources are images, the result surface should still work. A box of only images is valid. Operate reads what's there.

## Test Scenarios

### Image-only box
- User uploads 5 aspirational photos with no text.
- Operate runs. Produces three operator sentences. Gradient should be 1–2. Trust floor L1.
- Result should be useful and honest, not dismissive.

### Mixed box
- User uploads 3 images and 2 text sources.
- Operate reads all 5. Image descriptions participate in compression.
- Shape distribution reflects both text and image sources.

### Text-primary image
- User uploads a photo of a handwritten note.
- System routes to OCR, not visual description.
- Result .md contains the transcribed text, not an image description.

### Image with metadata
- User uploads a geotagged photo.
- System assigns L2 (exhibit — verifiable metadata exists).
- Description includes "GPS coordinates present" or similar factual note.

### Aspirational image with no verifiable content
- User uploads concept art or a stock photo.
- System assigns L1 and shape △ or ○.
- Does not fabricate provenance or location.
