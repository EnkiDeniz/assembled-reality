# Lœgos Box Self-Evolution Session Runbook

**Date:** April 7, 2026  
**Status:** Current-product operator runbook  
**Audience:** Founder/operator testing the live tool as it exists now  
**Purpose:** Take a fresh report about the product, put it into the correct preinstalled Lœgos box, use the current UI to evolve that box's seed, and leave the session with more convergence, more coherence, and a clearer next move.

## One-Sentence Answer

Yes: the current product is already capable of this loop, but the important truth is that the report does not evolve the box merely by being imported. The box evolves when you import the report as a source, inspect it, move useful lines or Seven findings into staging, shape the seed from that staged material, run Operate, inspect the contradictions or gaps, and then draft or seal a receipt.

## What This Session Is Really Doing

You are not using the report as a final answer. You are using it as a new witness.

The report enters the box as a source. Then you use the current product to decide which parts of that source deserve to affect the live seed.

That means the current runtime has four real moments:

1. Enter the right box.
2. Add the report as a source and read it raw.
3. Move the useful parts of that report into staging and shape the seed.
4. Run Operate, inspect what now holds or breaks, and close with proof.

## Current UX Truth

The current product already has the right primitives for this exercise:

1. Login and disclaimer are clear.
2. The box chooser is real and the seeded Lœgos origin box exists.
3. Source intake supports Markdown, pasted text, uploaded files, and links.
4. The Founder shell is a strong source-reading surface.
5. Seven can answer against the current document and move useful replies into staging.
6. The seed can be reshaped from staged material.
7. Operate, inspect, attestation, receipt drafting, and sealing all exist.

The main friction in the current journey is one seam:

The product is very good at showing you a source and very good at showing you proof, but the bridge between those two is still staging. If you do not understand staging, the journey can feel more mysterious than it should.

That is why this runbook is explicit about every click.

## Screen 10. Turn The Report Into Candidate Seed Material

At this point, the box has the report, but the seed has not changed yet.

You now choose one of two real current-product paths.

### Path A: Fastest current path

Use Seven to extract the most useful seed candidates from the report.

### Path B: More manual path

Carry specific lines from the report into staging yourself.

For most sessions, Path A is faster and cleaner. Path B is better when you want tighter human control over exactly what enters the seed.

## Screen 11A. Fastest Path: Ask Seven About The Report

**What you do:**  
Click `Ask Seven`.

Then use this prompt:

`Read this report as a UX and user-flow audit of the current Lœgos product. Extract the 5 smallest concrete seed updates that would improve the box "How Lœgos Assembled Itself". Write them as short declarative blocks grounded in the report, not marketing.`

If you want a more box-specific version, use this:

`Read this report as a new witness inside "How Lœgos Assembled Itself". What 3 to 7 new seed blocks should enter the box so its own seed better reflects the actual current user journey, friction points, and next-move logic?`

**What happens next:**  
Seven replies in the side panel. Each assistant reply has an `Add to staging` button.

**What you do next:**  
Click `Add to staging` on the Seven reply that best converts the report into useful seed material.

**What happens after that:**  
The report has now produced staged candidate blocks. The box has not fully changed yet, but the path to seed evolution has begun.

## Screen 11B. Manual Path: Carry Lines Directly From The Report

**What you do:**  
If you want to stage exact lines from the report rather than Seven's synthesis, click `Open full workspace`.

In the full workspace document view, click into important lines of the report and use the block action `Stage into weld`.

If you click the same line again, the action becomes `Remove from weld`.

Good manual candidates to stage from this report are:

1. The sentence explaining that the report is a source, not the seed.
2. The sentence explaining that the box evolves only when staging leads to seed shaping.
3. The lines naming the main UX seam around staging.
4. The lines identifying the actual shortest operator loop.
5. The lines describing what a successful close-out looks like.

**What happens next:**  
Those selected report blocks enter the seed queue directly.

**Why this matters:**  
This is the more faithful path if you want the seed to inherit the report's actual wording rather than Seven's interpretation of it.

## Screen 12. Move From Source Into Seed

**What you do:**  
Return to the source-first flow if needed, then click `Next: Shape seed`.

**What you see next:**  
The create-phase seed screen. It includes:

1. A `Next step` banner saying `Shape the first seed.`
2. A `Back to source` button when you came from the source-first path
3. The `Seed` header
4. The title `Shape the seed.`
5. Stage and Operate controls

**What matters here:**  
If you already added report-derived material to staging, this screen is now ready to turn that material into a seed update.

If you did not add anything to staging yet, the main action will effectively tell you so. That is a real current UX truth, not a mistake in this runbook.

## Screen 13. Open The Stage Panel

**What you do:**  
Click `Stage`.

**What you see:**  
The staging panel separates two different things:

1. `Seven findings`
2. `Seed queue`

If you staged a Seven reply, you may first see it under `Seven findings`.

**What you do next:**  
If the material is still under `Seven findings`, click `Add all` or the `+` button on the specific line you want.

That moves it into the actual `Seed queue`.

If you staged report lines manually, they will already be in `Seed queue`.

Inside `Seed queue`, you can:

1. Reorder with `↑` and `↓`
2. Remove with `−`
3. Clear the whole queue with `Clear`
4. Commit the queued material with `Assemble`

**Why this matters:**  
This is the most important current-product distinction in the whole session:

`Seven findings` are not yet the seed.  
`Seed queue` is the actual handoff surface.  
`Assemble` is what updates the seed.

## Screen 14. Shape The Seed

**What you do:**  
When the `Seed queue` contains the right report-derived lines, click `Assemble`.

**What happens next:**  
The current seed is updated or a new working seed document is created from the staged material. The product also drafts a receipt as part of the assembly flow.

**What changes in the box:**  

1. The report remains inside the box as a source.
2. The seed now includes material derived from that report.
3. The box has actually evolved, not just been annotated.

**Why this matters:**  
This is the exact moment when the report stops being external commentary and starts becoming internal structure.

## Screen 15. Run Operate On The Updated Seed

**What you do:**  
From the create or workbench view, run Operate.

Depending on the surface you are in, you will do this in one of two equivalent ways:

1. Click `Run Operate` from the top workspace control surface.
2. Use `Run inline Operate` from the diagnostics rail under `1. Seal preflight`.

**What happens next:**  
The product computes a fresh read of the current seed and opens the inspectable Operate state.

**Why this matters:**  
This is where the session becomes honest. The box now has to answer whether the report-derived seed changes actually improved clarity and convergence, or whether they simply added more text.

## Screen 16. Inspect What Broke, What Held, And What Needs Human Judgment

**What you see:**  
In the diagnostics and line inspection flow you now have access to:

1. Blocking contradictions
2. Shape and parse issues
3. Inline Operate findings
4. Summary Operate
5. Trust / depth / settlement

At the block level, you can inspect individual findings with `Inspect`.

**What you do:**  
Inspect the most consequential blocker or amber line, not just the first one.

Ask:

1. Did the report-derived seed blocks clarify the box's actual next move?
2. Did they introduce overclaiming?
3. Did they create a stronger relation between source, seed, and proof?

If a finding is correct, keep it.  
If a line is grounded in human judgment but missing enough machine-readable witness, use attestation carefully.

## Screen 17. Attestation, Only If Needed

**What you do:**  
If a line should remain because you, the human, judge it to be valid despite missing or partial machine evidence, use the `Attested override` area.

Enter a note and click `Attest block`.

**What happens next:**  
That line becomes explicitly human-attested rather than silently waved through.

**Why this matters:**  
This preserves the product's first principle. The machine can infer, but the human remains the interpreter of record.

Do not use attestation to hide a real contradiction. Use it only to record a human judgment the system cannot yet fully witness.

## Screen 18. Draft And Seal Proof

**What you see:**  
In the compiled-output / receipt area you have:

1. `Open receipts`
2. `Draft receipt`
3. `Seal latest`

**What you do:**  
Click `Draft receipt` if the current state has produced a meaningful improvement in clarity or convergence.

If the draft reflects a real turning point and you are ready to commit it as the current proof state, click `Seal latest`.

**What happens next:**  
The box records that this session was not just exploratory. It now has a proof artifact tied to the changed seed state.

**Why this matters:**  
Without this step, the session may still have been useful, but it remains softer and easier to lose. Drafting and sealing is what turns the session into a durable box event.

## Screen 19. Return To A Calm Summary Surface

**What you do:**  
Open the current box menu in the top control surface and click `Box overview`.

**What you should now see:**  

1. The imported report is present in `Recent sources`.
2. The proof surface reflects the new draft or sealed receipt.
3. The box metrics have moved from where they were before the session.
4. The box's next move is easier to name than it was before.

**Why this matters:**  
This is the correct place to judge session quality. Not inside the heat of staging, and not inside the excitement of Seven's answer. The calm test is whether box home now feels more coherent.

## What A Successful Close Looks Like

You should end the session able to say all of the following:

1. I put the report inside the correct box as a real source.
2. I did not confuse the source with the seed.
3. I converted the useful parts of the report into staged seed material.
4. I reshaped the seed from that material.
5. I ran Operate on the changed state.
6. I inspected at least one important blocker or ambiguity.
7. I drafted or sealed proof.
8. I left with a clearer next move than I had at the beginning.

If all eight are true, the session worked.

## The Best Current Test Prompt Sequence

If you want one literal testing sequence to use with this document inside the tool, use this order:

1. Import the report into `How Lœgos Assembled Itself`.
2. Read three to five lines in Founder shell and inspect them in `Lœgos read`.
3. Ask Seven:
   `Read this as a UX audit of the current product. What 5 seed updates should enter this box so the box better reflects the actual user journey?`

6. Open `Stage`.
7. Move the best lines from `Seven findings` into `Seed queue`.
8. Reorder them until they read like the smallest honest update.
9. Click `Assemble`.
10. Run `Operate`.
11. Inspect the strongest blocker or amber line.
12. Use `Attest block` only if human judgment genuinely outruns local witness.
13. Click `Draft receipt`.
14. If the state feels truly ready, click `Seal latest`.
15. Return to `Box overview` and judge the whole session from there.

## What This Journey Reveals About The Current UX

This exercise is valuable because it surfaces the product's real strengths and its real seams.

### What is strong right now

1. The intro and disclaimer establish the right seriousness.
2. The seeded Lœgos origin box is the correct object for self-referential testing.
3. Founder shell is a strong source-intake and reading surface.
4. Seven plus staging is a real mechanism for turning reflection into seed material.
5. Operate, inspect, attestation, and proof already form a meaningful close-out loop.

### What still feels rough

1. Box home does not let you add a source directly, so the operator has to know to resume first.
2. The bridge from source-reading to seed mutation is still too dependent on staging literacy.
3. `Next: Shape seed` is correct, but if nothing is staged yet the create screen can feel like a dead end.
4. The product is clearer about reading and proof than it is about the middle act of shaping.

### The single biggest opportunity

If this session feels more awkward than it should, the reason is not that the product lacks the necessary objects. The reason is that the seed-evolution handoff is still cognitively expensive.

The current shortest honest loop is:

`report as source -> inspect -> Ask Seven or stage lines -> Seed queue -> Assemble -> Operate -> Inspect -> Draft/Seal -> Box overview`

That loop is already real. It just needs to feel more inevitable.

## Final Operator Standard

The session succeeds if, at the end, the box does not merely contain a report about itself.

It succeeds if the box has:

1. one more real source,
2. a changed seed,
3. a fresh Operate read,
4. a visible proof event,
5. and a cleaner next move than it had before you started.

That is the correct test for whether Lœgos can use a document about its own user experience to further assemble itself.
