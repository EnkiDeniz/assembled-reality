## 1. Why You’re Getting This Memo

The product center has narrowed, and the design work needs to narrow with it.

For this MVP, we are not designing the whole imagined Lœgos system.
We are designing:

- `Room`
- `Dream`
- the seam between them

That means your job right now is not to design every future noun in the repo.
Your job is to make the smallest correct product feel coherent, calm, and unmistakably Lœgos.

---

## 2. What You Own In This Phase

You own the runtime surface design for the MVP.

That means:

1. the shell
2. `Room`
3. `Dream`
4. the `Bridge` interaction seam
5. the visual role of `workingEcho`
6. mobile and desktop behavior

You do **not** need to design:

- Recon as a runtime mode
- Drive Tape as a runtime mode
- Seven Terminal
- Shape Library as a primary destination
- box/session/witness/operate/mirror as permanent visible regions

Those may return later if they earn it.

---

## 3. Current MVP Rule

The MVP is not “chat plus some tools.”

The MVP stays Lœgos only if:

- `Room` feels like a live steering surface, not generic chat
- `Dream` feels like source contact, not storage
- `Bridge` feels like intentional carryover, not export plumbing
- `workingEcho` remains visible enough to steer by
- the shell stays quiet enough that the product reads as two durable surfaces, not many equal destinations

Your design work is what makes those truths legible.

---

## 4. What We Need From You

Please come back with a **proposal and plan** for the MVP surface design.

That response should cover five things.

### 4.1 Shell proposal

We need your recommendation for the smallest correct signed-in shell.

Specifically:

- how `Room | Dream` should appear
- how utility should stay subordinate
- how navigation should feel on mobile and desktop
- how the shell avoids becoming a control wall

The key standard:

**the shell is not the product center**

### 4.2 Room proposal

We need your recommendation for the MVP Room surface.

Specifically:

- active thread identity
- thread layout
- composer
- source entry
- `workingEcho`
- field-state visibility

The key standard:

**Room is not just chat; it becomes Lœgos through a thin but real workingEcho.**

### 4.3 Dream proposal

We need your recommendation for the MVP Dream surface.

Specifically:

- document access
- active document stage
- `Compiler Read`
- `Send to Room`
- the sequencing of listening

The key standard:

**Dream must first feel solid as document contact and source pressure.**

Listening matters, but it should not dominate the critical path before:

- the document stage is right
- Compiler Read feels honest and integrated
- the Dream -> Room handoff is clear

### 4.4 Bridge proposal

We need your recommendation for how the seam should feel in the UI.

Specifically:

- what a Dream -> Room handoff looks like
- how provenance should remain visible but not noisy
- how a user can feel continuity between Dream and Room
- how to avoid making the bridge feel like export, duplication, or side effect

The key standard:

**Bridge is a first-class seam, not a hidden technical step.**

### 4.5 Design sequencing

We need your judgment on what design decisions must be made first.

Current team alignment is:

1. Room
2. Dream document stage + Compiler Read + Send to Room
3. Bridge hardening
4. Listening polish

If you think a different order is needed for coherence or usability, say so directly.

---

## 5. What Not To Spend Time On

Please do **not** spend your main energy right now on:

- extra top-level destinations
- explaining ontology in the UI
- speculative future runtime modes
- audio polish before document-pressure clarity
- visual complexity that outruns the MVP
- a shell that treats every valid concept as a peer surface

This phase is about focus, not completeness.

---

## 6. Questions Your Response Should Answer

Please structure your response around these questions:

1. What is the minimum shell that still feels intentional?
2. What is the minimum Room that still feels like Lœgos instead of chat?
3. What is the minimum Dream that still feels like a reason to use the product on its own?
4. How should Bridge feel in the UI so Dream and Room remain one product?
5. What should remain visually quiet, inspectable, or deferred?
6. What design risks do you see if the team drifts back toward dashboard or multi-mode thinking?
