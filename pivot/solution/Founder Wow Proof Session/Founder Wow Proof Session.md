# Founder Wow Proof Session

## Purpose

Use this session to prove the de-obfuscation moment on a real repo document without switching into a fake demo path.

This is the founder check:

- does the text change in a way that feels meaningfully different from a plain document view
- can the user tell why a finding was flagged without guessing
- does the finding feel like a compiler-style read rather than a generic summary

## Standard Box

Use one dedicated local proof box and keep it separate from normal working boxes.

- Box title: `Founder Wow Proof`

If that box already exists, reuse it. Do not recycle a live working box for this session.

## Flow

1. Open or create the `Founder Wow Proof` box.
2. From the calm starter, choose `Add source`.
3. Upload the primary markdown file through the real markdown/source intake path.
4. Confirm the imported source opens in the raw source view, not the dense box/runtime.
5. Read the text, try listen, and confirm `Next: Shape seed` is the obvious next move.
6. Use `Next: Shape seed` to enter the real seed flow.
7. Ensure a live seed exists and open the current seed document as the active workbench surface.
8. Run inline Operate.
9. Let the app auto-focus the first meaningful finding.
10. Read the selected block, the inline evidence teaser, and the synchronized inspect panel together.

## Expected Visible States

- [ ] The app starts from the calm starter rather than the dense box/runtime.
- [ ] The imported source opens in a dedicated source view with raw text clearly visible.
- [ ] The source view keeps playback controls on the same screen.
- [ ] `Next: Shape seed` is visible and understandable without guessing.
- [ ] The app lands on the current seed before the inline findings matter.
- [ ] At least one block is auto-selected after Operate completes.
- [ ] The selected block is scrolled into view.
- [ ] The same finding is selected in the diagnostics rail.
- [ ] The selected block shows a short inline evidence teaser.
- [ ] The inspect panel gives enough rationale that the user does not have to guess why the block was flagged.

## Wow Checks

- [ ] At least one finding feels meaningfully different from the raw text.
- [ ] The user can explain why the finding was flagged without guessing.
- [ ] The finding feels like a real compiler-style read, not a generic summary.

Recommended evidence of success:

- [ ] an unsupported numeric or structural claim is called out
- [ ] vague consensus or assertion language is called out
- [ ] a grounded fact shows attached evidence
- [ ] a contradiction or missing bridge is made visible
